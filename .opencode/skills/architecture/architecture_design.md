# Skill: Architecture Design & Review
## 適用條件
- 當需要產出系統架構圖表（如流程圖、元件圖、部署圖）時
- 當需要評估現有或規劃中的架構設計時
- 當架構師需要驗證設計是否符合最佳實踐與專案約束時
- 當需要在任務規劃文件中加入架構視覺化說明時

## 輸入
- 架構描述文字（可選）：以自然語言或結構化要點描述的系統架構
- 專案上下文：EA_trade_bot 專案相關資訊（例如：MT5 整合、Docker 部署、交易策略模組等）
- 特定圖表類型（可選）：flowchart（流程圖）、class diagram（類別圖）、sequence diagram（序列圖）、state diagram（狀態圖）、deployment diagram（部署圖）、component diagram（元件圖）等（預設：依描述自動判斷）
- 評估重點（可選）：可擴展性、容錯性、效能、安全性、可維護性、MT5 整合正確性等（預設：全面評估）

## 輸出
- Mermaid 語法圖表程式碼（可直接複製到 .md 檔案中渲染）
- 架構評估報告：包括符合的最佳實踐、潛在問題與改進建議
- 決策依據摘要：針對關鍵架構決策的 Trade-off 說明
- 更新建議：如何將此架構設計納入 TaskPlan.md 或 DeveloperPrompt.md

## 執行步驟

### 第一步：需求澄清與範圍確定
1. 若輸入為自然語言描述，將其轉換為結構化架構要點：
   - 識別主要元件（例如：MT5 API 包裝層、策略引擎、風險管理模組、資料儲存層、介面層等）
   - 識別元件間的關係與資料流向
   - 識別外部系統或服務（例如：交易所 API、行情服務、警報服務等）
2. 確定要生成的圖表類型（基於描述焦點或明確指定）：
   - 流程圖：展示交易決策或資料處理流程
   - 元件圖：展示系統主要模組及其介面
   - 序列圖：展示關鍵情境下的元件互動（例如：下單流程）
   - 部署圖：展示 Docker 容器、主機、網路拓撲
   - 狀態圖：展示策略或訂單的生命週期狀態

### 第二步：產出 Mermaid 圖表語法
根據架構要點生成對應的 Mermaid 語法：
- **流程圖範例**：
  ```mermaid
  flowchart TD
    A[開始] --> B{市場數據到達?}
    B -->|是| C[策略演算運算]
    C --> D{產生交易訊號?}
    D -->|是| E[風險檢查]
    E -->|通過| F[下單至 MT5]
    F --> G[更新持倉狀態]
    G --> H[結束]
    D -->|否| H
    B -->|否| H
  ```
- **元件圖範例**：
  ```mermaid
  graph LR
    subgraph 前端
      UI[使用者介面]
    end
    subgraph 後端服務
      API[策略 API]
      ENGINE[策略引擎]
      RISK[風險管理]
      DATA[行情處理]
    end
    subgraph 基礎設施
      MT5[MT5 連線層]
      DB[(時序資料庫)]
    end
    UI --> API
    API --> ENGINE
    ENGINE --> RISK
    ENGINE --> DATA
    DATA --> DB
    RISK --> MT5
    MT5 -->|下單/查詢| 交易所[交易所 API]
  ```
- **部署圖範例**（Docker 佈署）：
  ```mermaid
  graph TD
    subgraph 主機機器[宿主機]
      direction TB
      subgraph Docker 網路[docker_network]
        STRATEGY[策略服務容器]
        DATA[行情服務容器]
        MT5_BRIDGE[MT5 連線橋接容器]
        ALERT[警報服務容器]
      end
    end
    MT5_BRIDGE -->|透過橋接網路| MT5終端機[MT5 終端機]
    STRATEGY -->|內部網路通訊| DATA
    STRATEGY -->|內部網路通訊| MT5_BRIDGE
    交易所[交易所 API] -->|網路| DATA
    交易所 -->|網路| MT5終端機
  ```

### 第三步：架構設計評估與最佳實踐檢查
對照以下檢查清單評估架構：
- **模組化與耦合度**：
  - 是否每個元件有單一職責？
  - 元件間是否透過明確介面溝通（避免直接資料庫或記憶體共享）？
  - 是否可以獨立開發、測試與部署？
- **效能與延遲考量（特別對交易系統）**：
  - 關鍵路徑（行情→策略→風險→下單）是否盡量同步且在同一進程或低延遲 IPC 中？
  - 是否避免了不必要的序列化/反序列化或網路跳躍？
  - 是否考慮了批次處理與緩衝以提高吞吐而不犧牲延遲？
- **錯誤處理與容錯性**：
  - 每個外部介面（MT5、交易所 API、資料庫）是否有重試與斷路器機制？
  - 是否有健康檢查與自動重啟機制？
  - 部分元件故障時，系統是否能安全降級或保持不交易狀態？
- **MT5 整合正確性**：
  - MT5 連線層是否獨立封裝，避免其他元件直接依賴 MT5 特有 API？
  - 是否考慮了 MT5 的單執行緒限制與事件循環？
  - 是否妥善處理了同步與異步呼叫的差異？
- **可觀測性**：
  - 是否包含了關鍵指標（延遲、錯誤率、吞吐）的暴露介面？
  - 是否有結構化日誌與追蹤 ID 傳播？
  - 是否規劃了警報閾值與通知管道？
- **可部署性與環境一致性**：
  - 是否所有依賴都透過 Dockerfile 或 docker-compose 明確說明？
  - 是否區分了開發、測試與生產環境的配置？
  - 是否考慮了配置與密鑰的外部化（例如：環境變數或 Vault）？
- **交易特有風險控制**：
  - 風險管理模組是否在下單前最後一道關卡？
  - 是否包含了倉位限制、每日虧損上限、間隔時間等硬性限制？
  - 是否有機制可以在緊急情況下全盤停止所有策略？

### 第四步：產出評估報告與建議
1. 列出通過的最佳實踐項目
2. 列出不符合或需改進的項目，並提供：
   - 問題描述
   - 可能的影響
   - 具體改進建議（可包含參考架構或模式）
3. 若有多種設計選項，提供 Trade-off 分析表（例如：效能 vs. 複雜度，開發時間 vs. 長期維護等）
4. 給出優先級建議（高/中/低）以及預估工作量

## 品質標準
- 產出的 Mermaid 語法必須是有效的，可在支援 Mermaid 的 Markdown 渲染器中正確顯示
- 評估報告必須具體、可操作，避免空泛的「良好」或「需改進」描述
- 建議必須基於公認的架構模式（例如：微服務、事件驅動、分層架構、六邊形架構等）或交易系統的已知最佳實踐
- 所有建議都必須說明為何對 EA_trade_bot 這個特別系統有幫助（例如：低延遲需求、MT5 特殊限制等）
- 若無法從輸入中推斷足夠資訊進行圖表生成或評估，必須明確說明需要哪些額外資訊

## 使用範例
```
我將啟用 architecture_design skill來為EA_trade_bot的策略引擎與風險管理模組產生架構圖表並進行設計評估。
輸入描述：
EA_trade_bot 包含三個主要服務：市場數據處理服務（透過 WebSocket 接收行情）、策略引擎服務（根據技術指標產生交易訊號）、風險管理服務（訂單前最後檢查與倉位控制）。
市場數據經過初步處理後傳遞給策略引擎，策略引擎產生訊號後送風險管理進行檢查，通過後才下單至 MT5。
所有服務皆以 Docker 容器運行，使用內部 Docker 網路進行 gRPC 通訊。
MT5 連線由一個專門的橋接容器負責，以避免直接阻塞主要服務執行緒。
需要評估此架構在低延遲與容錯性方面的表現，並提供改進建議。
```
