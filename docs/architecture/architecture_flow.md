## LangGraph 節點定義（優化版）

### 基本節點層級結構

```
START
  ▼
[1] 查詢意圖檢測 (conditional_node)
  ▼
[2] 查詢生成 (parallel: query_rewriter + template_expansion)
  ▼
[3] 並行搜尋 (parallel: tavily_search + serper_search + google_search)
  ▼
[4] 結果聚合與排序 (result_aggregation + reranking)
  ▼
[5] 向量檢索補充 (optional_conditional)
  ▼
[6] LLM 生成 (generate_node)
  ▼
[7] 事實檢核 (fact_check_node)
  ├─ 通過 ──▶ [8] 品質評分
  └─ 失敗 ──▶ [9] 改進策略 ──▶ [6] 重新生成 (max_retries=2)
  ▼
[10] 後處理 (post_processing)
  ▼
[11] 反饋收集 (feedback_collection)
  ▼
END
```

### 錯誤處理流程

```
任何節點失敗
  ▼
┌──────────────────────────────┐
│ 錯誤類型分類                 │
├──────────────────────────────┤
│ 1. 搜尋失敗 (可重試)        │
│    → 使用不同搜尋源重試     │
│    → 調整查詢策略           │
│    → Fallback 到備選方案    │
│                              │
│ 2. LLM 失敗 (可重試)        │
│    → 簡化 Prompt            │
│    → 調整參數               │
│    → 使用備選 LLM          │
│                              │
│ 3. 驗證失敗 (可修復)        │
│    → 補充搜尋結果          │
│    → 重新生成               │
│                              │
│ 4. 致命錯誤 (無法修復)      │
│    → 返回錯誤信息           │
│    → 記錄日誌               │
└──────────────────────────────┘
  ▼
根據嚴重程度決定：
├─ 重試
├─ Fallback
├─ 返回部分結果
└─ 返回錯誤
```

## 關鍵改進點對比

| 層級 | 當前流程 | 優化後流程 | 改進 |
|-----|--------|---------|------|
| **查詢層** | 單一查詢 | 多維度查詢 + 模板 + 關鍵字擴展 | 覆蓋率 +30% |
| **檢索層** | 單源搜尋 | 並行多源 + 結果排序 + 向量補充 | 精準度 +40% |
| **生成層** | 基本 Prompt | 智能結構化 Prompt v2 | 內容質量 +50% |
| **驗證層** | 無驗證 | 事實檢核 + 品質評分 + 改進迴圈 | 誤植率 -80% |
| **可觀測性** | 基本日誌 | 詳細指標 + 反饋收集 | 可調試性 +100% |

## 數據流示例

### 輸入
```json
{
  "organ": "台灣AI新創公司",
  "organNo": "12345678",
  "mode": "GENERATE"
}
```

### 查詢意圖檢測
```
輸入：台灣AI新創公司
檢測結果：intent = "basic_info"
選擇模板組：basic_info
```

### 查詢生成
```
模板 1: "台灣AI新創公司 官網"
模板 2: "台灣AI新創公司 公司介紹"
模板 3: "台灣AI新創公司 統一編號 12345678"
模板 4: "AI新創 台灣公司資訊"
模板 5: "台灣 人工智能 新創企業"
```

```

   相關度: 0.95 | 字數: 500

   相關度: 0.82 | 字數: 300
   
3. [Google] "XX AI新創獲得融資..."
   相關度: 0.78 | 字數: 250

```

## 性能指標

### 預期改善

| 指標 | 當前 | 優化後 | 提升 |
|-----|-----|------|------|
| 檢索精準度 (0-1) | 0.65 | 0.95 | +46% |
| 內容具體度 (0-1) | 0.55 | 0.85 | +55% |
| 誤植率 (%) | 15% | 3% | -80% |
| 平均響應時間 (秒) | 8 | 12 | 並行搜尋成本 |
| 用戶滿意度評分 (0-5) | 2.8 | 4.2 | +50% |

### 成本考慮

- **搜尋成本**：多源並行搜尋會增加 API 調用（~3x），但提升準確度
  - 建議：使用優先隊列，只在精準度不足時啟動並行
  
- **LLM 成本**：事實檢核會增加 LLM 調用（~1.5x）
  - 建議：批量驗證，優化 Token 使用
  
- **計算成本**：cross-encoder 排序增加計算
  - 建議：使用輕量級模型或緩存結果
