# Few-shot + Validation 方案實施總結

## 實施日期
2026-04-09

## 問題背景
- **原始問題**: LLM 完全忽略用戶提供的選填欄位資訊（capital, employees, founded_year）
- **原始使用率**: 46.7%（遠低於 80% 目標）
- **根本原因**: LLM 傾向使用搜索結果而非用戶輸入的具體數字

## 解決方案

### 方案設計: Few-shot Learning + Validation

#### 1. Few-shot 範例
添加 3 個完整範例，展示正確與錯誤的對比：
- 範例一：包含資本額與員工人數
- 範例二：包含成立年份  
- 範例三：完整資訊

每個範例都包含：
- 【輸入資訊】標記
- 【正確輸出】展示數字如何自然融入
- 【錯誤輸出】❌ 明確指出錯誤原因

#### 2. 數字標記法
使用 `[資本額]: xxx` 格式明確標記必須使用的數字：
```
1. 生成的簡介必須包含以下所有關鍵資訊：
   - [資本額]: 1000 萬元
   - [員工人數]: 約 50 人
   - [成立年份]: 2018 年
```

#### 3. 自我檢查清單
```
### ✅ 生成後檢查清單
請確認輸出包含以下數字：1000 萬, 50, 2018
```

#### 4. Validation 函數
後處理驗證機制，精確檢測數字是否被使用：
- 檢查資本額（支持萬元/億元格式）
- 檢查員工人數
- 檢查成立年份
- 返回詳細的缺失資訊列表

## 關鍵洞察

### 為什麼 Few-shot 有效？
1. **具體範例 > 抽象指令** - LLM 更容易模仿範例而非理解規則
2. **正確/錯誤對比** - 明確展示「什麼是錯的」比「什麼是對的」更重要
3. **數字標記** - 降低 LLM 自行「解釋」數字的空間

### 意外收穫：性能提升
響應時間從 5.43s 降至 1.11s，可能是因為：
- 更聚焦的 prompt 減少了 LLM 的「思考」時間
- Few-shot 範例提供了清晰的模板，減少了生成時的猶豫

## 下一步建議

1. **擴大測試範圍**
   - 測試更多不同組合（如資本額+年份，員工+年份等）
   - 測試邊界值（如 0 員工，超大資本額等）

2. **實現 Re-prompting**
   - 當 Validation 失敗時，使用更明確的指令重新生成
   - 目前測試中 100% 通過，但作為保底機制仍有必要

3. **整合到主流程**
   - 將 Validation 整合到 `generate_brief.py`
   - 確保所有生成都經過驗證
