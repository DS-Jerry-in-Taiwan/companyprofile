# 公司簡介生成工作流程分析報告

基於對現有實現的分析和HackMD URI-B-04廠商Detail API的參考，以下是公司簡介生成API的完整使用流程分析：

## 一、現有實現流程分析

### 1. 前端觸發公司簡介生成的時機和條件
- **觸發時機**：透過POST請求到 `/v1/company/profile/process` 端點
- **觸發條件**：
  - 必須提供 `organNo` (廠商編號) 和 `organ` (廠商名稱)
  - `mode` 參數必須為 `GENERATE` 或 `OPTIMIZE`
  - 在 `GENERATE` 模式下，`companyUrl` 為可選參數（系統會使用廠商名稱進行網路搜尋）

### 2. API請求的具體參數
- **必要參數**：
  - `organNo`: 廠商編號 (字串)
  - `organ`: 廠商名稱 (字串)
  - `mode`: 操作模式，必須為 "GENERATE" 或 "OPTIMIZE"
- **可選參數**：
  - `companyUrl`: 廠商網站URL (在GENERATE模式下可選)
  - `brief`: 現有公司簡介 (在OPTIMIZE模式下必要)
  - `word_limit`: 字數限制 (50-2000之間的整數，預設未設定)
  - `optimization_mode`: 優化模式，必須為 "STANDARD", "CONCISE" 或 "DETAILED" (預設: "STANDARD")

### 3. 資科如何針對部分欄位上網搜尋
- 當 `mode` 為 `GENERATE` 且未提供 `companyUrl` 時：
  1. 在 `generate_brief.py` 中呼叫 `web_search(organ, company_url)`
  2. `web_search` 函數使用 Serper.dev API 進行搜尋
  3. 搜尋查詢格式為：`"{company_name} 官網"` (例如: "台達電 官網")
  4. 回傳前 N 個結果的 URL 列表 (預設 N=5)
  5. 取得第一個結果作為目標 URL 進行內容擷取

### 4. 是否成功找到廠商資料的判斷標準
- **成功條件**：
  1. 提供了 `companyUrl` 參數，或
  2. 透過 `web_search(organ, None)` 取得至少一個 URL 結果
- **失敗條件**：
  1. 未提供 `companyUrl` 且網路搜尋未返回任何結果
  2. 在此情況下，系統會拋出 `ExternalServiceError` 錯誤：
     ```
     'No company URL available. Please provide a companyUrl or ensure the organ name yields search results.'
     ```

### 5. 兩種生成路徑分析

#### 路徑一：成功找到資料（目前實現的主要路徑）
1. 取得目標 URL（來自搜尋結果或提供的 companyUrl）
2. 使用 `extract_main_content()` 從目標 URL 擷取主要內容
3. 使用 `preprocess_text()` 進行文字前處理（清理HTML、去除多餘空白等）
4. 使用 `build_generate_prompt()` 建構發送給 LLM 的 Prompt
5. 呼叫 `call_llm()` 取得 LLM 生成的公司簡介
6. 使用 `post_process()` 進行後處理（格式化、長度調整等）
7. 返回生成的公司簡介

#### 路徑二：未找到資料（目前實現中的缺失）
- **目前狀況**：系統不會嘗試此路徑，而是直接拋出錯誤
- **應該實現的路徑**：
  1. 當無法取得公司網站內容時，僅使用廠商名稱 (`organ`) 等基本資訊
  2. 建構一個基於有限資訊的 Prompt 發送給 LLM
  3. 由 LLM 基於公司名稱和常識生成公司簡介
  4. 返回生成的公司簡介（品質可能較低，但總比錯誤好）

### 6. 結果回傳給求才系統的流程
1. API 接收請求並進行參數驗證
2. 執行核心邏輯（生成或優化公司簡介）
3. 建構成功回應：
   ```json
   {
     "status": "success",
     "data": {
       "brief": "生成的公司簡介內容",
       "metadata": {
         "organNo": "...",
         "organ": "...",
         "generatedAt": "...",
         "processingTimeMs": ...
       }
     }
   }
   ```
4. 返回 HTTP 200 狀態碼和 JSON 回應
5. 同時記錄請求和回應的日誌（包含效能監控）

### 7. 錯誤處理和日誌記錄機制
- **錯誤類型**：
  - `ValidationError`: 輸入參數驗證失敗 (HTTP 400)
  - `ExternalServiceError`: 外部服務失敗 (網路搜尋、內容擷取等) (HTTP 500)
  - `LLMServiceError`: LLM 呼叫失敗 (HTTP 500)
  - 一般 `Exception`: 未預期的錯誤 (HTTP 500)
- **日誌機制**：
  - 結構化日誌記錄（包含 request_id、trace_id）
  - API 請求/回應日誌（含效能指標）
  - 錯誤和警告日誌
  - 異常偵測和報告機制
  - 日誌查詢端點 (`/v1/monitoring/logs`)
  - 異常統計端點 (`/v1/monitoring/anomalies`)

## 二、流程調整建議（基於目前實現的差距）

基於對 HackMD URI-B-04 API 的參考，目前實現存在以下主要差距：

### 差距 1：資料來源單一
- **目前狀況**：僅依賴公司網站內容擷取
- **問題**：許多公司可能沒有明顯的官網，或官網內容不足以生成專業簡介
- **建議**：整合多種資料來源，類似 HackMD API 提供的結構化公司資料

### 差距 2：缺乏備用生成路徑
- **目前狀況**：無法取得網站內容時直接失敗
- **問題**：用戶體驗不佳，即使只有基本公司資訊也應該能生成簡介
- **建議**：實施真正的備用路徑，在網路搜尋失敗時仍能基於有限資訊生成簡介

### 差距 3：資訊利用不完整
- **目前狀況**：主要使用網頁內容的文字，未充分利用結構化資料
- **問題**：未能利用公司基本資訊（成立年份、員工數、認證等）來豐富生成內容
- **建議**：設計更好的 Prompt，將結構化公司資訊融入生成過程

### 差距 4：缺少品質控制機制
- **目前狀況**：生成後直接返回，缺少事後驗證
- **問題**：無法確保生成的簡介符合品質標準或包含必要資訊
- **建議**：實施生成後品質檢查機制

## 三、框架建議（如何更好地結構這個功能）

建議採用模組化、管線化的架構設計：

### 建議架構：公司簡介生成管線 (Company Brief Generation Pipeline)

```
[輸入處理] → [資料採集器] → [資料融合器] → [Prompt 建構器] → [LLM 引擎] → [後處理器] → [品質檢查器] → [輸出格式化]
```

#### 1. 輸入處理器 (Input Processor)
- 負責參數驗證和標準化
- 處理不同輸入場景（GENERATE vs OPTIMIZE）
- 標準化公司識別資訊

#### 2. 資料採集器 (Data Collector) - 核心改進點
- **多來源採集策略**：
  - 首選：公司網站內容擷取（現有實現）
  - 次選：結構化公司資料 API（參考 HackMD URI-B-04）
  - 最後選擇：基於公司名稱的網路搜尋摘要
- **採集優先級**：結構化API > 網站內容 > 網路搜尋摘要
- **容錯機制**：任一來源失敗時自動切換到次優來源

#### 3. 資料融合器 (Data Fusion Engine)
- 將來自不同來源的資訊智慧融合
- 去除重複資訊
- 識別和標記資訊來源的可靠度
- 準備結構化的公司資料檔案供 Prompt 使用

#### 4. Prompt 建構器 (Prompt Builder)
- 根據可用資料豐富度動態調整 Prompt 策略
- 高品質資料：詳細指導 LLM 如何組織和優化內容
- 有限資料：提供更具體的結構和範例來引導 LLM
- 基於公司類型和產業特性調整語氣和重點

#### 5. LLM 引擎 (LLM Engine)
- 現有實現的核心部分，保持不變
- 可考慮加入快取機制避免重複請求相同公司

#### 6. 後處理器 (Post Processor)
- 現有實現，可加強：
  - 長度控制更精確
  - 專業術語標準化
  - 法規合規性檢查

#### 7. 品質檢查器 (Quality Checker) - 新增模組
- 必要元素檢查：是否包含公司成立時間、核心業務等關鍵資訊
- 語言品質檢查：流暢度、專業度、無明顯錯誤
- 長度適切性：符合指定的字數限制
- 事實一致性：與來源資料不矛盾

#### 8. 輸出格式化器 (Output Formatter)
- 標準化回應格式
- 加入處理元數據（使用的資料來源、處理時間等）
- 提供不同詳細程度的輸出選項

## 四、具體的改進方向

### 優先級 1：實施備用生成路徑（最高優先級）
- **目標**：確保API永遠不會因找不到網站內容而失敗
- **實施步驟**：
  1. 修改 `generate_brief.py`，在網路搜尋失敗時不拋錯誤
  2. 建立 fallback 路徑：僅使用 `organ` (公司名稱) 建構 Prompt
  3. 在 Prompt 中明確說明資訊有限，請 LLM 基於常識生成合理的公司簡介
  4. 在回應中加入品質指標標記（例如: `dataQuality: "limited"`）

### 優先級 2：整合結構化公司資料來源
- **目標**：提升生成簡介的準確性和專業度
- **實施步驟**：
  1. 評估並整合可用的公司資料API（如台灣公司註冊資料、商業登記資料等）
  2. 參考 HackMD URI-B-04 的資料結構設計適配器
  3. 建立公司資料服務層，統一不同來源的資料取得介面
  4. 在資料融合階段將結構化資料與網站內容結合

### 優先級 3：增強 Prompt 智慧
- **目標**：根據可用資料品質動態調整生成策略
- **實施步驟**：
  1. 建立資料豐富度評估機制（根據可用欄位數量和品質評分）
  2. 設計多種 Prompt 模板：
     - 高品質資料模板：著重於組織和優化現有資訊
     - 中等品質資料模板：融合多來源資訊並填補缺口
     - 低品質資料模板：提供詳細結構引導 LLM 創造合理內容
  3. 實作動態 Prompt 選擇邏輯

### 優先級 4：添加品質控制機制
- **目標**：確保輸出符合最低品質標準
- **實施步驟**：
  1. 建立公司簡介品質評估標準
  2. 實作自動品質檢查規則：
     - 必要資訊檢查（公司名稱、成立時間、業務類型等）
     - 語言品質基本檢查
     - 長度適切性檢查
  3. 對於不達標的結果，可選擇：
     - 自動重試（調整 Prompt 參數）
     - 標記品質問題並返回
     - 在可能的場合提供備用版本

### 優先級 5：優化錯誤處理和日誌
- **目標**：提升系統可觀測性和除錯能力
- **實施步驟**：
  1. 在資料採集階段加入更詳細的來源追蹤日誌
  2. 實作資料品質指標記錄（來源可靠度、資訊完整度等）
  3. 加入效能基準測試點追蹤
  4. 優化異常偵測規則以更好地識別系統問題

## 五、優先級排序

### 高優先級 (P0) - 必須實施
1. **實施備用生成路徑**：確保API穩定性，避免因單點失敗而影響服務
2. **優化錯誤處理**：使備用路徑的失敗也能被適當處理和記錄

### 中高優先級 (P1) - 应該實施
3. **整合結構化公司資料來源**：顯著提升生成品質的基礎設施
4. **增強 Prompt 智慧**：根據資料品質優化生成策略

### 中優先級 (P2) - 有實施價值
5. **添加品質控制機制**：確保輸出品質一致性
6. **優化監控和日誌**：提升系統可維護性

### 低優先級 (P3) - 可選實施
7. **實施快取機制**：提升重複請求的效能
8. **加入A/B測試框架**：支援不同Prompt策略的比較

## 六、結論

目前的公司簡介生成API實現了基本的功能流程，但在穩定性、資料利用效率和品質控制方面還有顯著改進空間。特別是缺乏備用生成路徑使系統在面對網路問題時過於脆弱。

參考 HackMD URI-B-04 API 的設計理念，建議採用多來源資料採集和智慧融合的架構，這不僅能提高系統的穩定性（透過備用路徑），還能顯著提升生成公司簡介的品質和專業度。

實施上述改進將使公司簡介生成API從一個簡單的網站內容包裝服務，轉變為一個真正智慧的公司資訊理解和生成平台。
