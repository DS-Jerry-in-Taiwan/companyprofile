# Phase 1 - 同步 MVP Checkpoint 協議

**階段**: `phase1_sync_mvp_dev`
**目的**: 定義在同步 MVP 開發過程中，人工介入審查和決策的觸發點、選項和流程。

## 🚦 Checkpoint 1 - 架構設計確認

### **觸發點**
*   @ARCH 完成 `API 介面定義` 和 `模組介面定義` 後。

### **審查內容**
*   **`docs/mvp_company_profile_api.yml`**:
    *   `POST /v1/company/profile/process` 的請求和響應模型是否清晰、合理。
    *   是否已移除或註解所有非同步相關的定義。
*   **模組介面定義**:
    *   各模組 (例如 `Request Validator`, `Web Search Service`, `LLM Service Wrapper`) 的類別/函數介面是否明確，職責是否單一。
    *   數據流和錯誤處理的規劃是否合理。

### **決策選項**
1.  **✅ 確認通過**: 架構設計符合預期，可進入 @CODER 實作階段。
2.  **🔍 詳細檢查**: 需要進一步審查具體細節 (例如，要求 @ARCH 提供更詳細的介面定義或數據流圖)。
3.  **❌ 問題：[描述]**: 發現設計缺陷或不合理之處，需要 @ARCH 進行修正。
4.  **🔄 重新執行 Phase 1.2**: 要求 @ARCH 重新進行架構設計。

## 🏁 Checkpoint 2 - 測試與功能驗證確認

### **觸發點**
*   @ANALYST 完成所有測試、功能驗證和 Demo 準備後。

### **審查內容**
*   **測試報告**:
    *   單元測試和整合測試的通過率。
    *   測試覆蓋率 (如果可衡量)。
*   **功能驗證結果**:
    *   GENERATE 模式是否能成功生成符合預期的簡介。
    *   OPTIMIZE 模式是否能成功優化簡介。
    *   HTML 消毒和內容過濾是否按預期工作。
    *   錯誤處理是否符合預期。
*   **Demo 演示**:
    *   API 功能是否能順利演示。

### **決策選項**
1.  **✅ 確認通過**: MVP 功能已實現，品質達標，可進行交付。
2.  **🔍 詳細檢查**: 需要進一步審查測試報告、LLM 輸出細節或 Demo 流程。
3.  **❌ 問題：[描述]**: 發現功能缺陷、測試未通過或輸出不符合預期，需要 @CODER 或 @ANALYST 進行修正。
4.  **🔄 重新執行 Phase 1.3/1.4**: 要求 @CODER 重新實作或 @ANALYST 重新測試。
