# 最簡同步 AI 公司簡介生成/優化服務 (MVP) - 高階模組架構規劃


**核心理念：** 將所有功能模組化，並在單一 HTTP 請求的生命週期內，以同步方式依序執行，直接返回結果。


#### **1. API Gateway / Controller (入口點)**


*   **職責：**
    *   接收所有來自客戶端的 HTTP 請求 (`POST /v1/company/profile/process`)。
    *   執行基本的請求解析 (JSON)。
    *   呼叫 `Request Validator` 進行輸入參數驗證。
    *   根據驗證結果，將請求分派給 `Core Logic Dispatcher`。
    *   處理來自下游模組的響應或錯誤，並將其格式化為 HTTP 響應返回給客戶端。
*   **關鍵組件：**
    *   HTTP Server (例如：Python Flask/FastAPI, Node.js Express, Go Gin)。
    *   路由定義。


#### **2. Request Validator (請求驗證器)**


*   **職責：**
    *   驗證輸入參數的完整性、類型和業務邏輯正確性。
    *   例如：`mode` 必須是 `GENERATE` 或 `OPTIMIZE`。
    *   `GENERATE` 模式下 `companyUrl` 必須存在且格式正確。
    *   `OPTIMIZE` 模式下 `brief` 必須存在且非空。
    *   `organNo` 和 `organ` 必須存在。
*   **輸出：**
    *   如果驗證失敗，拋出 `ValidationError` (由 API Controller 捕獲並返回 400)。
    *   如果驗證成功，返回乾淨、結構化的請求數據。


#### **3. Core Logic Dispatcher (核心邏輯分派器)**


*   **職責：**
    *   根據請求中的 `mode` 參數，將處理流程分派到 `Generate Brief Module` 或 `Optimize Brief Module`。
    *   這是整個業務邏輯的中央協調者。
*   **輸入：** 經過驗證的請求數據。
*   **輸出：** 最終的 `CompanyProfileResponse` 對象。


#### **4. Generate Brief Module (生成簡介模組)**


*   **職責：** 負責從零開始生成公司簡介。
*   **內部流程 (同步執行)：**
    1.  **Web Search Service (網路搜尋服務)：**
        *   **職責：** 根據 `organ` (公司名稱) 和 `companyUrl`，呼叫外部搜尋引擎 API (例如 Serper.dev) 獲取相關網頁連結。
        *   **輸入：** `organ`, `companyUrl`。
        *   **輸出：** 相關網頁 URL 列表。
    2.  **Web Scraper / Content Extractor (網頁爬蟲/內容提取器)：**
        *   **職責：** 遍歷 `Web Search Service` 返回的 URL 列表，同步發送 HTTP 請求獲取網頁內容，並提取主要文本。
        *   **輸入：** 網頁 URL 列表。
        *   **輸出：** 原始網頁文本內容列表。
    3.  **Text Preprocessor (文本預處理器)：**
        *   **職責：** 對提取的原始文本進行清洗、合併、去重、摘要，以準備 LLM 輸入。確保文本長度在 LLM 的 Context Window 限制內。
        *   **輸入：** 原始網頁文本內容列表。
        *   **輸出：** 供 LLM 使用的精煉上下文文本。
    4.  **Prompt Builder (生成模式 Prompt 構建器)：**
        *   **職責：** 根據 `organ`, `companyUrl`, `Text Preprocessor` 提供的上下文文本，構建用於 LLM 生成公司簡介的 Prompt。
        *   **輸入：** `organ`, `companyUrl`, 上下文文本。
        *   **輸出：** LLM Prompt 字串。
    5.  **LLM Service Wrapper (LLM 服務封裝器)：**
        *   **職責：** 呼叫 LLM 服務 (例如 Gemini, OpenAI)，並解析其結構化輸出。
        *   **輸入：** LLM Prompt。
        *   **輸出：** LLM 原始響應 (預期為 JSON 格式，包含 `body_html`, `summary`, `tags`)。
    6.  **Post-processing Module (後處理模組)：**
        *   **職責：** 對 LLM 的原始輸出進行安全和內容檢查。
        *   **輸入：** LLM 原始響應。
        *   **輸出：** 經過處理的 `body_html`, `summary`, `tags`。


#### **5. Optimize Brief Module (優化簡介模組)**


*   **職責：** 負責優化現有的公司簡介。
*   **內部流程 (同步執行)：**
    1.  **Prompt Builder (優化模式 Prompt 構建器)：**
        *   **職責：** 根據使用者提供的 `brief`, `organ`, `optimization_mode`, `trade`, `word_limit` 等參數，構建用於 LLM 優化公司簡介的 Prompt。
        *   **輸入：** `brief`, `organ`, `optimization_mode` 等。
        *   **輸出：** LLM Prompt 字串。
    2.  **LLM Service Wrapper (LLM 服務封裝器)：**
        *   **職責：** 呼叫 LLM 服務，並解析其結構化輸出。
        *   **輸入：** LLM Prompt。
        *   **輸出：** LLM 原始響應 (預期為 JSON 格式，包含 `body_html`, `summary`, `tags`)。
    3.  **Post-processing Module (後處理模組)：**
        *   **職責：** 對 LLM 的原始輸出進行安全和內容檢查。
        *   **輸入：** LLM 原始響應。
        *   **輸出：** 經過處理的 `body_html`, `summary`, `tags`。


#### **6. LLM Service Wrapper (LLM 服務封裝器)**


*   **職責：** 統一與 LLM 供應商 (例如 Gemini, OpenAI) 的互動介面，處理 API 金鑰、請求格式、響應解析、錯誤重試等。
*   **輸入：** LLM Prompt。
*   **輸出：** LLM 原始響應 (通常是 JSON 字串)。
*   **注意：** 這是 `Generate Brief Module` 和 `Optimize Brief Module` 共同依賴的底層服務。


#### **7. Post-processing Module (後處理模組)**


*   **職責：** 對 LLM 產出的內容進行最終的清理和驗證。
*   **內部流程：**
    1.  **HTML Sanitizer (HTML 消毒器)：**
        *   **職責：** 使用 `bleach` 等工具清理 LLM 輸出中的 HTML 標籤，防止 XSS 攻擊。
        *   **輸入：** `body_html`。
        *   **輸出：** 安全的 `body_html`。
    2.  **Content Filter (內容過濾器)：**
        *   **職責：** 檢查 `body_html`, `summary`, `tags` 中是否包含敏感詞、競品名稱等。
        *   **輸入：** `body_html`, `summary`, `tags`。
        *   **輸出：** 經過過濾的內容，或拋出 `ContentViolationError`。
*   **注意：** 這也是 `Generate Brief Module` 和 `Optimize Brief Module` 共同依賴的模組。


#### **8. Response Formatter (響應格式化器)**


*   **職責：** 將最終處理好的數據組合成符合 `CompanyProfileResponse` 結構的 JSON 響應。
*   **輸入：** 來自 `Generate Brief Module` 或 `Optimize Brief Module` 的處理結果。
*   **輸出：** 標準化的 JSON 響應。


#### **9. Error Handler (錯誤處理器)**


*   **職責：** 集中處理應用程式中發生的各種錯誤 (驗證錯誤、外部服務錯誤、LLM 錯誤、內容過濾錯誤等)。
*   **輸出：** 統一格式的錯誤響應 (例如 `ErrorResponse` 或 `ServerError`)，包含錯誤代碼和訊息，並映射到適當的 HTTP 狀態碼。
