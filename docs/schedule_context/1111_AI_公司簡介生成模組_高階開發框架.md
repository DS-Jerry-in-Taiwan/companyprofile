# 1111 人力銀行 - 公司簡介 AI 生成/優化模組：高階開發框架

## 1. 應用場景 (Application Scenarios)
*   **新廠註冊引導**：廠商新加入 1111 平台，尚未填寫簡介。系統自動根據「公司名稱」與「官網網址」產出專業簡介，降低入駐門檻。
*   **舊廠資料優化**：既有廠商的簡介過於簡略或資訊陳舊。透過 AI 抓取最新網路動態（如新聞、產品發佈）進行擴充與潤飾。
*   **招募吸引力提升**：針對缺乏品牌行銷能力的企業，將生硬的經營項目轉化為具有「雇主品牌」吸引力的專業文案。

## 2. 應用需求 (Application Requirements)
*   **功能性需求**：
    *   **多源輸入**：支援「公司名稱」、「參考 URL」及「原始簡介文字」作為 Context。
    *   **檢索增強 (RAG)**：整合外部搜尋（如 Google/Serper），解決 LLM 對非知名公司資訊不足（Hallucination）的問題。
    *   **雙模式處理**：支援「從零生成 (Generate)」與「結構優化 (Optimize)」兩種 Prompt 策略。
    *   **安全性檢查**：嚴格過濾博弈、詐騙、競品（104、yes123 等）相關文字與敏感詞。
*   **非功能性需求**：
    *   **高可用與非同步**：外部 API 調用（搜尋+LLM）耗時長，必須設計為非同步處理機制，避免 API 逾時。
    *   **輸出結構化**：確保 AI 回傳的 HTML 格式穩定，且符合 1111 前端編輯器的渲染規範。

## 3. 應用指標 (Application Metrics)
*   **採用率 (Save Rate)**：生成的簡介被用戶點擊「採納並儲存」的比例（目標 > 60%）。
*   **平均處理時長 (E2E Latency)**：從觸發到產出結果的完整時間（目標 < 20s）。
*   **合規通過率 (Compliance Rate)**：產出內容通過內部「高風險字眼檢查」API 的成功率（目標 100%）。
*   **幻覺率 (Hallucination Rate)**：隨機抽樣檢查生成內容中關鍵事實（如資本額、地址）的準確性。

## 4. 應用流程 (Application Process)
1.  **觸發階段**：前端 B 端後台發送請求（包含公司名稱及 URL）。
2.  **檢索階段**：後端 Worker 觸發搜尋引擎 API，抓取 3-5 筆相關網頁資訊。
3.  **清洗階段**：提取網頁主體內容，過濾無效的 Token（廣告、Nav 等）。
4.  **推理階段**：將清洗後的資訊 + 廠商輸入 + System Prompt 送入 LLM 進行推理。
5.  **後置階段**：對 AI 回傳內容進行「風險詞過濾」及「HTML 消毒」。
6.  **交付階段**：更新 Job 狀態，前端透過回調或輪詢呈現結果供廠商修改或確認。

## 5. 應用架構 (Application Architecture)
*   **Presentation Layer**：1111 B 端後台（React/Angular），負責渲染 AI 處理狀態與結果。
*   **Service Layer (Core API)**：
    *   **Task Orchestrator**：負責 API 路由、權限校驗與非同步任務（Queue）派發。
    *   **Risk Control Engine**：介接內部敏感詞檢查 API 與競品過濾器。
*   **Intelligence Layer (AI Worker)**：
    *   **Search Agent**：串接 Serper.dev / Google Search API。
    *   **LLM Wrapper**：封裝 Gemini / OpenAI SDK，負責 Prompt 注入與 Token 管理。
*   **Data Layer**：
    *   **Redis**：儲存 Job 狀態與臨時生成的快取。
    *   **PostgreSQL**：儲存最終由用戶核可的公司簡介正式資料。
