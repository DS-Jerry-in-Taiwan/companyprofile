## 1. Phase 28 — 儲存層中央集權 + 雙軌儲存（地端 SQLite、雲端 DynamoDB）

### 1.1 背景與問題

**問題一：重複初始化**
`llm_service.py`、`error_logger.py`、`api_controller.py`、`services/llm_service.py` 四個檔案各自初始化自己的 SQLite 連線，一個請求進來 SQLite 被初始化 4 次。

**問題二：Lambda 上 SQLite 唯讀**
Lambda 檔案系統唯讀，SQLite 寫入全部在 catch block 中靜默失敗。儲存層等於不存在。

### 1.2 解決方案

儲存層依環境自動切換：**地端寫 SQLite、雲端寫 DynamoDB**，所有 consumer 透過 `src.storage.get_storage()` 拿同一個 instance，不關心底層是誰。

### 1.3 實作內容

**Phase A — 全域入口 (`src/storage/__init__.py`)**
- 新增 `init_storage(config)` — 強制傳入 config，由高階位置決定建立 SQLite 還是 DynamoDB
- 新增 `get_storage()` — 所有 consumer 統一入口
- 新增 `is_initialized()` — 檢查狀態
- 5 項測試

**Phase B — DynamoDB Adapter + Factory 擴充**
- 新增 `src/storage/dynamodb_adapter.py`：實作 `StorageInterface`（save_response、get_response、list_by_organ、save_error、list_errors）
- `StorageFactory.create()` 新增 `type=dynamodb` 分支
- `config/storage_config.json` 新增 lambda-dev / lambda-prod profiles
- 14 項測試（使用 unittest.mock 模擬 boto3）
- 修復 `datetime.utcnow()` deprecation warning

**Phase C — Consumer 重構（4 個檔案）**
- `api_controller.py`：加入環境判斷（`STAGE=dev/prod → DynamoDB`，其他 → SQLite）+ `init_storage()`；移除 `_save_failed_request_to_llm_responses()`（功能已被 ErrorLogger + call_llm 涵蓋）
- `error_logger.py`：`_init_storage()` 改 call `get_storage()`
- `utils/llm_service.py`：`_get_storage()` 改 call `get_storage()`
- `services/llm_service.py`：`_get_storage()` 改 call `get_storage()`

**Phase D — serverless.yml DynamoDB + IAM**
- 新增 `LlmResponsesTable`（PK: trace_id, SK: created_at, GSI: organ_no, status）
- 新增 `ErrorLogsTable`（PK: trace_id, SK: created_at, GSI: error_code）
- IAM 權限：`PutItem`、`GetItem`、`Query`、`Scan`（限兩張 Table + GSI）
- 既有 ALB / Target Group 完全未動

**Phase E — sqlalchemy 確認**
- 確認 `sqlalchemy` 不在任何 requirements.txt 中，無需處理

**Bug Fix — 既有測試**
- 修復 `test_sqlite_adapter.py` 與 `test_storage_integration.py` 中 `request_id` → `trace_id`（欄位已於 Phase 24 改名，測試未跟著更新）
