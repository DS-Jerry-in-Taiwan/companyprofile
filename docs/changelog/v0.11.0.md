# v0.11.0 — LLM 輸出品質架構

**發布日期**: 2026-05-12

---

## 新增功能

### Phase 39: 三層防護架構

#### 第一層：Prompt 分離（預防 LLM 誤解）

- **移除自檢指令**: `prompt_builder.py` 中「生成後請自行檢查字數」「✅ 檢查點」「✅ 生成後自我檢查」全部移除，改為單純的約束條件
- **加入 system_instruction**: 透過 API 的 `system_instruction` 參數傳遞行為規則（「只輸出最終結果，不要輸出推理過程」），與 user prompt 分離
- **system_instruction 參數化**: 從寫死在 provider 改為由 `llm_service.py` 傳遞參數，provider 保持通用

#### 第二層：API 層強制 JSON 格式

- **response_schema**: Gemini API 加入 `response_mime_type="application/json"` + `response_schema`，從 `LLMOutput` Pydantic model 動態產生，消除 raw text fallback 路徑
- **JSON 解析分支**: Gemini 直接 `json.loads`，Bedrock 保留 `_parse_response()` 向後相容

#### 第三層：品質閘門 + 重試機制

- **8 項品質檢查**（固定條件，不隨 LLM 調整）：
  - `check_company_name` — title/body/summary 任一含公司名
  - `check_min_length` — body_html ≥ 100 字
  - `check_normal_start` — 開頭為中文字元（非英文/數字/符號）
  - `check_no_truncation` — 不以 `...` 結尾
  - `check_no_artifact_markers` — 不含 `**`、`【正確輸出】`
  - `check_no_repetition` — 無重複句子（LLM 迴圈檢測）
  - `check_no_template_leftovers` — 無 `{var}` 模板變數殘留
  - `check_chinese_ratio` — 中文字元比例 ≥ 30%
- **重試機制**: RetryManager 控制最多 2 次重試，重試時關閉隨機多樣化（固定模板參數）
- **重試用盡回傳警語**: 不回異常內容，改回傳「暫時無法產生簡介，請稍後再試」

#### 後處理 Pipeline 化

- **Processor 抽象架構**: `src/processors/` 模組，提供 Processor 抽象類別、註冊查找、Pipeline 執行器
- **11 個獨立 Processor**: 每個處理步驟獨立檔案（html_sanitizer、markdown_cleaner、verbose_remover、taiwan_converter 等）
- **配置驅動**: `config/post_processing.json` 控制 Processor 啟用與執行順序，可動態開關
- **向後相容**: `post_process()` 函數簽名不變，內部改走 Pipeline

## 架構調整

### 三層防護流（取代原本的單層後處理清洗）

```
原本:
  LLM(自由文字) → _parse_response() → post_process(monolith清洗)
                  失敗→raw text當內容 → VERBOSE_PATTERNS無限擴充

現在:
  system_instruction(行為規則) + prompt(內容素材)
    → Gemini API(response_schema 強制 JSON)
    → QualityGate(8 項檢查)
      ├─ 通過 → Pipeline(配置驅動 11 個 Processor) → 輸出
      └─ 失敗 → Retry(最多 2 次) → 用盡 → 回傳警語
```

### 新新增模組

| 模組 | 路徑 | 說明 |
|------|------|------|
| Processors | `src/processors/` | Processor 抽象 + 11 個實作 + Pipeline 執行器 |
| Quality | `src/quality/` | QualityGate + 8 項品質檢查 |
| Retry | `src/retry/` | RetryManager 重試管理器 |

### 配置檔新增

| 配置檔 | 說明 |
|--------|------|
| `config/post_processing.json` | 控制 11 個 Processor 的啟用與執行順序 |

## 已知問題

- Bedrock Provider 不支援 `response_schema`，JSON 解析保留 `_parse_response()`，但不影響品質閘門
- DETAILED 模式在搜尋階段可能逾時（與本階段無關）
- QualityGate 的 threshold 值（100 字、0.3 比例）目前寫死在程式碼中，可考慮後續配置化

## 後續修正（Phase 40）

- `check_company_name` 已從 QualityGate 移除——該檢查屬於品質評估而非異常檢測，且對含內部註記的公司名產生誤擋
- `retry_info` 已新增 `quality_logs` 表持久化，不再僅存在 CloudWatch logs
